# SOLID → 소유권 결정 번역 표

SOLID 5원칙을 "누가 무엇을 소유하고 무엇으로 가리키는가"의 판단 기준으로 번역한다.
각 항목: 원칙의 소유권 의미 → 점검 질문 → 위반 신호 → (예시).

> 예시는 독립 DataSet ↔ Grid 케이스로 든다 — 어디까지나 *예시*이며 이 스킬의 대상이 아니다.

## SRP — 한 조각 = 한 Owner

- **소유권 의미**: 모듈이 바뀔 이유는 하나. 그 이유는 보통 "어떤 상태조각의 SSOT인가"로 환원된다.
- **점검**: 각 상태조각에 단일 Owner 모듈이 있나? 한 모듈이 둘 이상 조각의 SSOT인가?
- **위반 신호**: 한 조각을 두 모듈이 각자 보관(동기화 코드 발생), 또는 주인 없는 떠도는 상태.
- **결정**: 둘 이상 SSOT면 분리 후보. 주인 없으면 Owner를 명시 지정.
- **예시**: DataSet=행+CRUD 상태 SSOT, Grid=선택·활성셀 SSOT. 행을 둘 다 소유하면 SRP 위반.

## OCP — 새 소비자는 구독으로 확장

- **소유권 의미**: 새 소비자(다른 Grid/Form/뷰)를 추가할 때 Owner를 수정하지 않고 확장 가능한가.
- **점검**: 소비자 추가 = Owner의 공개 구독/조회 surface 사용만으로 되나? Owner 내부를 고쳐야 하나?
- **위반 신호**: 소비자 추가마다 Owner에 분기·콜백·필드가 늘어남.
- **결정**: 변경 통지를 구독(이벤트/시그널)으로 노출. 소비자별 로직은 소비자 쪽에.
- **예시**: DataSet은 `subscribe`만 노출하고, 새 Grid가 붙어도 DataSet 코드는 그대로.

## LSP — Identity key 계약 일관성

- **소유권 의미**: 같은 정체성 키를 모든 소비자가 *동일 의미*로 쓴다.
- **점검**: ID 발급·재발급·삭제 후 의미가 모든 소비자에게 같나? 어떤 소비자만 다른 규칙을 가정하나?
- **위반 신호**: 한 소비자는 ID를 영속으로, 다른 소비자는 세션 한정으로 가정.
- **결정**: 키의 수명·재발급 규칙을 한곳에서 못박고 모든 소비자가 따른다.
- **예시**: RowId는 load마다 재발급. 이를 영속 키로 오해한 소비자가 있으면 LSP 위반.

## ISP — 소비자별 surface 분리

- **소유권 의미**: 소비자는 자기가 쓰는 것만 본다. 읽기 전용 소비자에 명령·변경 이벤트를 강요하지 않음.
- **점검**: 읽기 surface와 명령 surface가 분리됐나? 안 쓰는 명령·이벤트까지 한 인터페이스에 묶였나?
- **위반 신호**: 표시 전용 뷰가 CRUD 명령 전체를 가진 객체에 의존.
- **결정**: 조회 surface와 변경 surface를 나눠 필요한 쪽만 주입.
- **예시**: Grid 표시 경로는 `getRows`/구독만, 편집 경로만 `setValue`를 본다.

## DIP — 참조 방향은 추상에 의존

- **소유권 의미**: 경계를 넘는 의존은 구체→구체가 아니라 추상(인터페이스/주입)을 향한다. 경계 넘는 식별은 ID로만.
- **점검**: 참조 화살표가 한 방향인가? 순환인가? 구체 타입을 직접 잡나, 주입된 계약을 잡나?
- **위반 신호**: Owner가 소비자를 역참조, 또는 경계 넘어 내부 객체/index를 직접 들고 다님.
- **결정**: 방향을 한쪽으로 고정. 경계 넘는 참조는 ID(값)로, 동작은 주입된 port로.
- **예시**: Grid→DataSet 단방향. Grid는 DataSet 내부 배열 index가 아니라 RowId(값)로 행을 가리킴.

## 종합 출력

5항목 점검 결과를 표로: `원칙 | 통과/마찰 | 근거(조각·모듈) | 결정·후보`.
마찰은 분리(SRP/ISP) 또는 방향 역전(DIP) 또는 키 계약 고정(LSP) 후보로 연결한다.
