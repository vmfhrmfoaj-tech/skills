# `design-feature` 스킬 생성 계획

## 요약

상세 요구사항을 구현 가능한 설계로 변환하는 범용 `design-feature` 스킬을 생성한다. 특정 기술이나 프레임워크에 종속되지 않고 프로젝트의 패러다임에 맞춰 설계하며, 클래스나 패턴을 억지로 도입하지 않는다. 최종 산출물에는 SOLID 검토, Mermaid 다이어그램과 TDD 구현 전략을 포함한다.

완성된 스킬은 Codex가 자동으로 발견하도록 `${CODEX_HOME:-~/.codex}/skills/design-feature`에 설치한다.

## 스킬 구성

- `SKILL.md`: 발동 조건, 의존 스킬 검사, 아키텍처 분석, 책임 결정 대화, SOLID 검토, 상세 설계, 다이어그램과 TDD 계획 절차를 정의한다.
- `agents/openai.yaml`: 표시 이름, 짧은 설명과 `$design-feature`를 명시적으로 호출하는 기본 프롬프트를 정의한다.
- `references/design-output-template.md`: 최종 Markdown 산출물 계약과 재사용할 Mermaid·표 템플릿을 정의한다.
- 워크플로가 추론과 저장소 탐색 중심이므로 스크립트와 에셋은 추가하지 않는다.

`$improve-codebase-architecture`, `$grill-with-docs`, `$tdd`를 필수 설치 스킬로 취급한다. 누락된 스킬이 있으면 이를 흉내 내거나 우회하지 않고, 누락 목록과 설치 필요성을 알린 뒤 중단한다.

## 실행 계약

1. 질문하기 전에 요구사항, 소스, 테스트, 설정, 용어집과 ADR을 탐색한다.
2. `$improve-codebase-architecture`를 호출해 영향받는 모듈, 인터페이스, 심(seam), 어댑터, 결합도, 지역성, 레버리지와 책임 후보를 분석한다.
3. SOLID, 인터페이스 깊이, 삭제 테스트와 테스트 용이성으로 책임 후보를 평가한다. 프로젝트 패러다임에 적합한 경우에만 클래스를 사용한다.
4. 공개 계약, 도메인·상태 소유권, 의존성 방향, 실패 의미, 호환성 또는 향후 변경 비용에 영향을 주는 의미 있는 선택에는 `$grill-with-docs`를 호출한다.
   - 한 번에 하나의 질문만 한다.
   - 현실적인 선택지, 결과와 명확한 추천안을 제시한다.
   - 문서 존재 여부가 아니라 설계 선택의 중요도를 발동 기준으로 삼는다.
   - 명시적 사용자 승인 없이 `CONTEXT.md`, ADR이나 다른 파일을 수정하지 않고, 필요한 문서 변경은 최종 설계에 제안한다.
5. 합의된 책임을 인터페이스, 모듈, 타입, 어댑터와 구체적인 파일 작업에 연결한다.
6. Mermaid 구조·의존성 다이어그램을 항상 포함하고, 여러 모듈에 걸친 실행 순서가 중요하면 시퀀스 다이어그램을 추가한다.
7. 계약이 안정된 뒤 `$tdd`를 호출해 수직 Red–Green–Refactor 슬라이스, 공개 동작 테스트, 테스트 수준, 실제 테스트 더블 심과 우선순위가 있는 실패 사례를 작성한다.
8. 기본적으로 완결된 Markdown을 채팅으로 반환하고 사용자가 명시적으로 요청한 경우에만 파일로 저장한다.

## 최종 산출물 계약

최종 설계는 목표와 범위, 현재 아키텍처, 변경 영향, 소유권 결정, 공개 계약, 타입과 데이터 흐름, 모듈·파일 책임, 적용 가능한 SOLID 근거, 실패·호환성 동작, 다이어그램, 요구사항-테스트 추적성과 TDD 구현 순서를 포함한다.

객체 지향 시스템에는 클래스·인터페이스, 함수형 시스템에는 모듈·함수·데이터 타입, UI 시스템에는 상태 소유자·액션·리소스·어댑터를 사용한다. 혼합 구조가 가장 작고 명확한 설계라면 근거를 밝히고 조합한다.

## 검증

- 완성된 스킬에 `quick_validate.py`를 실행한다.
- TypeScript/SolidJS 사례로 타입, 상태 소유권, 파일과 다이어그램의 상세도를 확인한다.
- 비-TypeScript 사례로 기술 중립성을 확인한다.
- 책임 충돌 사례에서 추천을 포함한 한-질문씩 대화가 시작되는지 확인한다.
- 되돌리기 쉬운 국소 배치 결정에는 불필요한 질문이 생기지 않는지 확인한다.
- 문서 충돌, 필수 스킬 누락, 실패 흐름, 조건부 시퀀스 다이어그램과 무단 파일 수정 방지를 확인한다.
- 모든 요구사항이 소유자, 공개 동작과 TDD 시나리오에 연결되는지 확인한다.

## 확정된 기본값

- 스킬명은 `design-feature`다.
- 아키텍처 분석 후 SOLID 책임 할당과 결정 대화를 진행한다.
- `$grill-with-docs`를 책임·소유권 결정 대화의 중심 스킬로 사용한다.
- 구조 다이어그램은 필수이며 행위 시퀀스 다이어그램은 조건부다.
- 채팅 Markdown이 기본 산출물이며 파일 변경에는 명시적 승인이 필요하다.
